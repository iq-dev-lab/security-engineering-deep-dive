# 공격자 관점 전환 — STRIDE 위협 모델

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 방어자 관점과 공격자 관점은 구체적으로 무엇이 다른가?
- STRIDE 모델의 6가지 위협 범주는 무엇이고, Spring 서비스에서 어떻게 나타나는가?
- 내 시스템에서 가장 위험한 진입점을 어떻게 찾는가?
- "보안 설정을 켰다"는 것과 "공격 표면을 줄였다"는 것은 어떻게 다른가?

---

## 🔍 왜 이 사고방식이 실무에서 중요한가

2023년 한 핀테크 스타트업이 침해 사고를 당했다. 사후 분석 결과 Spring Security는 정상적으로 활성화되어 있었고, JWT 인증도 동작 중이었다. 취약점은 `/api/admin/users/{id}` 엔드포인트에 `@PreAuthorize("hasRole('USER')")`가 누락된 것이었다. 공격자는 일반 사용자 토큰으로 전체 사용자 목록을 수집하고 이틀 동안 데이터를 조용히 유출했다.

개발팀은 "Spring Security를 설정했으니 안전하다"고 믿었다. 공격자는 "Spring Security가 설정되어 있더라도 개별 엔드포인트에 권한 체크가 누락된 경우가 많다"고 알고 있었다.

이것이 **방어자 관점**과 **공격자 관점**의 차이다.

방어자는 자신이 설정한 것을 기준으로 생각한다. 공격자는 설정되지 않은 것, 빠진 것, 가정이 깨지는 경우를 기준으로 생각한다.

## 😱 흔한 실수 (Before — 공격자 관점 없이 설계할 때)

```
시나리오: 새 API 엔드포인트 설계

개발자의 관점:
  "이 API는 인증된 사용자만 호출할 수 있다"
  → JWT 토큰 검증 필터 적용 → 완료

놓친 질문들:
  - 인증된 사용자가 다른 사용자의 데이터를 조회할 수 있는가? (IDOR)
  - 토큰에 포함된 role 클레임을 서버에서 재검증하는가?
  - 요청 빈도에 제한이 있는가? (브루트포스)
  - 에러 응답이 내부 구조를 노출하는가?
  - 이 API가 내부 서비스를 호출한다면, 해당 URL을 공격자가 조작할 수 있는가? (SSRF)

결과:
  JWT 인증은 통과하지만 위 취약점 중 하나로 침해 발생
  "Spring Security가 있는데 왜 뚫렸지?" 라는 질문을 하게 됨
```

---

## ✨ 올바른 접근 (After — 공격자 관점으로 설계할 때)

```
같은 API 엔드포인트를 공격자 관점으로 검토:

질문 1: 이 엔드포인트로 인증 없이 접근할 수 있는가?
  → Security Filter Chain에서 이 경로가 permitAll()에 포함되어 있지 않은지 확인

질문 2: 인증된 사용자가 다른 사용자의 데이터에 접근할 수 있는가?
  → PathVariable의 리소스 소유자 검증 코드 추가

질문 3: 악의적인 입력값으로 내부 동작을 조작할 수 있는가?
  → 입력값 화이트리스트 검증, SQL/명령어 인젝션 방어 적용

질문 4: 이 엔드포인트를 통해 시스템 정보를 추출할 수 있는가?
  → 에러 응답 표준화(스택 트레이스 미포함), 정보 최소 노출

질문 5: 반복 호출로 피해를 줄 수 있는가?
  → Rate Limiting 적용
```

## 🔬 STRIDE 위협 모델 — 6가지 위협 범주

STRIDE는 Microsoft가 개발한 위협 분류 체계로, 소프트웨어 시스템에서 발생할 수 있는 위협을 6가지로 분류한다. 각 범주는 Spring 백엔드 서비스에서 구체적인 형태로 나타난다.

### S — Spoofing (신원 위장)

**정의**: 다른 사용자나 시스템인 척 행세하는 공격

**Spring 서비스에서의 형태**:
```
- JWT 토큰 위조 (alg:none 공격, 서명 키 추측)
- 세션 ID 탈취 후 재사용
- OAuth2 state 파라미터 미검증으로 인한 토큰 탈취
- API Key를 추측하거나 유출된 키로 인증 우회
```

**방어 질문**: "이 시스템은 요청자가 실제로 주장하는 당사자임을 어떻게 검증하는가?"

---

### T — Tampering (데이터 변조)

**정의**: 전송 중이거나 저장된 데이터를 무단으로 수정하는 공격

**Spring 서비스에서의 형태**:
```
- SQL Injection으로 DB 데이터 수정
- HTTP 요청의 파라미터 변조 (가격, 수량, 사용자 ID)
- JWT 페이로드 변조 (role 클레임 상승)
- CSRF로 인증된 사용자를 대신하여 상태 변경 요청 실행
```

**방어 질문**: "이 시스템의 데이터가 전송 중 또는 처리 중에 변조된다면 어떻게 감지하는가?"

---

### R — Repudiation (부인)

**정의**: 수행한 행위를 나중에 부인할 수 있는 상황

**Spring 서비스에서의 형태**:
```
- 로그가 없어 누가 언제 어떤 데이터를 조회/수정했는지 추적 불가
- 감사 로그가 변조 가능한 저장소에 보관됨
- 트랜잭션 실행 증거가 없어 분쟁 발생 시 입증 불가
```

**방어 질문**: "민감한 행위에 대해 나중에 '내가 하지 않았다'는 주장을 반박할 수 있는 증거를 남기는가?"

---

### I — Information Disclosure (정보 노출)

**정의**: 권한 없는 사용자에게 정보가 노출되는 것

**Spring 서비스에서의 형태**:
```
- 에러 응답에 스택 트레이스, DB 쿼리, 내부 경로 포함
- Spring Actuator /actuator/env 프로덕션 노출
- 로그에 비밀번호, 카드번호, 개인정보 기록
- IDOR로 다른 사용자 데이터 조회
- application.properties가 Git에 평문으로 커밋됨
```

**방어 질문**: "이 시스템에서 권한 없는 사용자가 접근할 수 있는 정보는 무엇인가?"

---

### D — Denial of Service (서비스 거부)

**정의**: 정상 사용자가 서비스를 이용하지 못하게 만드는 공격

**Spring 서비스에서의 형태**:
```
- Rate Limiting 없는 API에 대한 대량 요청
- 계정 잠금 정책을 악용한 정상 사용자 계정 잠금
- 대용량 파일 업로드로 디스크/메모리 고갈
- 복잡한 정규식(ReDoS)이나 깊은 JSON 중첩으로 CPU 고갈
- Slow HTTP 공격으로 연결 풀 고갈
```

**방어 질문**: "공격자가 이 시스템을 정상 사용자가 이용하지 못하게 만들 수 있는 방법은 무엇인가?"

---

### E — Elevation of Privilege (권한 상승)

**정의**: 허가된 것보다 높은 권한을 획득하는 공격

**Spring 서비스에서의 형태**:
```
- Mass Assignment로 role: "ADMIN" 주입
- @PreAuthorize 누락된 엔드포인트로 관리자 기능 호출
- JWT roles 클레임을 서버에서 재검증하지 않아 권한 상승
- 디렉토리 트래버설로 설정 파일 접근
- SQL Injection으로 DB 관리자 권한 실행
```

**방어 질문**: "일반 사용자가 관리자 기능을 실행할 수 있는 경로가 있는가?"

---

## 💻 실전 적용 — Spring 서비스에 STRIDE 적용하기

다음은 간단한 주문 API에 STRIDE를 적용하는 예시다.

```
대상: POST /api/orders (주문 생성 API)

S - Spoofing 검토:
  ✓ JWT 토큰 검증 중인가?
  ✓ 토큰의 subject(사용자 ID)와 요청 본문의 userId가 일치하는지 검증하는가?
  ✗ 알고리즘 화이트리스트 없이 JWT 파싱 → alg:none 취약점

T - Tampering 검토:
  ✓ 가격, 할인율은 클라이언트가 아닌 서버에서 계산하는가?
  ✗ 요청 본문의 price 필드를 그대로 사용 → 가격 변조 가능

R - Repudiation 검토:
  ✗ 주문 생성 시 사용자 ID, IP, 타임스탬프 로깅 없음
  ✗ 주문 취소 이력이 남지 않음

I - Information Disclosure 검토:
  ✗ 재고 부족 에러에 현재 재고 수량 포함 → 내부 정보 노출
  ✗ 존재하지 않는 상품 ID로 요청 시 DB 에러 메시지 그대로 반환

D - Denial of Service 검토:
  ✗ Rate Limiting 없음 → 초당 수천 건 주문 생성 가능
  ✗ 매우 긴 주소 입력값 길이 제한 없음

E - Elevation of Privilege 검토:
  ✓ 일반 사용자는 자신의 주문만 생성 가능 (userId 검증)
  ✗ 주문 상태 변경 API에 @PreAuthorize 누락 → 사용자가 배송 상태 변경 가능
```

## 📊 공격자 관점 체크리스트

새 API를 설계하거나 리뷰할 때 다음 질문을 순서대로 점검한다.

```
인증 (Authentication)
  □ 인증 없이 이 엔드포인트에 접근할 수 있는가?
  □ 인증 토큰의 서명과 만료를 검증하는가?
  □ 알고리즘 혼동 공격이 가능한가?

인가 (Authorization)
  □ 인증된 사용자가 다른 사용자의 리소스에 접근할 수 있는가? (수평)
  □ 일반 사용자가 관리자 기능을 호출할 수 있는가? (수직)
  □ 리소스의 소유자를 검증하는가?

입력값 검증
  □ 모든 외부 입력값은 화이트리스트로 검증되는가?
  □ SQL/명령어/LDAP 인젝션이 가능한가?
  □ 입력값의 크기/형식 제한이 있는가?

정보 노출
  □ 에러 응답이 내부 정보를 포함하는가?
  □ 로그에 민감한 정보가 기록되는가?
  □ 응답 헤더가 서버/프레임워크 버전을 노출하는가?

가용성
  □ Rate Limiting이 적용되어 있는가?
  □ 대용량 입력으로 리소스를 고갈시킬 수 있는가?

감사
  □ 민감한 행위에 대한 로그가 남는가?
  □ 로그는 변조되지 않는 저장소에 보관되는가?
```

## ⚖️ 트레이드오프

| 항목 | 공격자 관점 도입 | 미도입 |
|------|-----------------|--------|
| 설계 시간 | 엔드포인트당 10~30분 추가 | 없음 |
| 발견 시점 | 설계/개발 단계 | 운영 중 침해 후 |
| 수정 비용 | 낮음 (코드 수정) | 높음 (데이터 유출, 법적 대응, 신뢰 손실) |
| 코드 복잡도 | 권한 검증, Rate Limiting 등 추가 | 낮음 |
| 팀 학습 비용 | 초기 1~2주 | 없음 |

보안 검토는 기능 개발 후에 추가하는 것이 아니라 설계 단계에서 함께 수행할 때 비용이 가장 낮다. 출시 후 발견된 취약점의 수정 비용은 설계 단계 수정 비용의 평균 30배다 (NIST 연구).

## 📌 핵심 정리

- **공격자는 빠진 것을 찾는다** — 방어자는 설정한 것을 기준으로 생각하지만, 공격자는 설정되지 않은 것, 누락된 것, 가정이 깨지는 경우를 찾는다
- **STRIDE는 질문 도구다** — 완벽한 체크리스트가 아니라 "어떤 종류의 위협이 있을 수 있는가?"를 생각하게 만드는 프레임워크다
- **모든 신뢰 경계를 점검하라** — 외부 요청뿐 아니라 내부 서비스 간 호출, 관리자 API, 배치 작업도 공격 표면이다
- **Spring Security는 시작점이다** — 프레임워크가 기본 방어를 제공하지만, 각 엔드포인트의 비즈니스 로직 수준 인가는 개발자 책임이다

## 🤔 생각해볼 문제

**문제 1**: 다음 Spring 컨트롤러에서 STRIDE 각 범주에 해당하는 취약점을 찾아라.

```java
@RestController
@RequestMapping("/api/transfers")
public class TransferController {
    
    @PostMapping
    public ResponseEntity<Transfer> createTransfer(
            @RequestBody TransferRequest request) {
        
        User fromUser = userRepository.findById(request.getFromUserId())
            .orElseThrow(() -> new RuntimeException("User not found: " + request.getFromUserId()));
        
        if (fromUser.getBalance() < request.getAmount()) {
            throw new RuntimeException("Insufficient balance. Current: " + fromUser.getBalance());
        }
        
        Transfer transfer = transferService.execute(request);
        return ResponseEntity.ok(transfer);
    }
}
```

**해설**:
```
S (Spoofing): 요청자가 실제 fromUserId의 소유자인지 검증하지 않음
  → 다른 사람의 계좌에서 이체 실행 가능
  
T (Tampering): amount 값을 음수로 보내면 잔액이 증가할 수 있음
  → 비즈니스 검증(amount > 0) 누락
  
R (Repudiation): 이체 실행 로그(실행자, IP, 타임스탬프) 없음
  → "나는 이체를 요청한 적 없다" 주장 시 반박 불가
  
I (Information Disclosure): 에러 메시지에 실제 잔액 노출
  → 공격자가 다른 사용자의 잔액 추측 가능
  
D (Denial of Service): Rate Limiting 없음
  → 잔액 확인 반복 요청으로 서비스 부하
  
E (Elevation of Privilege): @PreAuthorize 없음
  → 인증 없이 호출 가능 (Security Config에 따라)
```

**문제 2**: "JWT를 사용하면 CSRF를 걱정하지 않아도 된다"는 주장이 어떤 조건에서 맞고, 어떤 조건에서 틀린가?

**해설**:
```
맞는 조건:
  - JWT를 Authorization 헤더(Bearer)로만 전송하는 경우
  - 브라우저가 자동으로 헤더를 첨부하지 않으므로 CSRF 불가

틀린 조건:
  - JWT를 쿠키에 저장하는 경우 (httpOnly 쿠키라도)
  - 브라우저가 쿠키를 자동 첨부 → CSRF 공격 가능
  - SameSite 쿠키 속성이 없거나 None으로 설정된 경우

결론:
  저장 위치(쿠키 vs 로컬스토리지)와 전송 방식(자동 vs 수동)이
  CSRF 위협 여부를 결정한다. "JWT를 쓴다"는 것 자체는 CSRF와 무관하다.
```
