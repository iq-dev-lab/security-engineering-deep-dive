# 위협 모델링 방법론 — DFD와 PASTA

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- DFD(Data Flow Diagram)로 공격 표면을 어떻게 시각화하는가?
- DFD의 신뢰 경계(Trust Boundary)가 왜 위협 모델링의 핵심인가?
- PASTA 방법론의 7단계는 무엇이고, 기존 체크리스트 방식과 어떻게 다른가?
- 위험 우선순위를 어떻게 정량적으로 평가하는가?
- 스프린트 주기에 위협 모델링을 어떻게 통합하는가?

---

## 🔍 왜 위협 모델링이 실무에서 중요한가

대부분의 보안 사고는 운영 중에 발견된다. 그러나 침해 사고 원인의 70% 이상은 설계 단계에서 이미 결정된 구조적 취약점이다. 코드 레벨의 버그(SQL Injection, XSS 등)는 패치로 수정할 수 있지만, 잘못된 신뢰 관계 설계나 권한 모델은 전체 아키텍처를 바꿔야 한다.

위협 모델링은 **설계 단계에서 공격자 관점을 체계적으로 도입**하는 방법론이다. STRIDE가 "어떤 종류의 위협이 있는가?"라는 분류 도구라면, DFD와 PASTA는 "어디서, 어떤 순서로, 얼마나 심각한 위협이 있는가?"를 찾아내는 프로세스다.

## 😱 흔한 실수 (Before — 체크리스트 방식의 한계)

```
전형적인 보안 검토 방식:
  설계 완료 → 코드 작성 → 배포 → 보안팀 검토
  
보안팀 체크리스트:
  ✓ HTTPS 사용?
  ✓ 인증 있음?
  ✓ SQL Injection 방어?
  → "통과"

실제로 놓친 것:
  - 내부 마이크로서비스 간 통신이 인증 없이 HTTP로 동작
  - 관리자 API가 VPN 없이 인터넷에 직접 노출
  - 배치 서비스가 운영 DB에 읽기/쓰기 모두 가능한 계정 사용
  - 사용자 업로드 파일이 퍼블릭 S3 버킷에 저장됨

문제:
  체크리스트는 알려진 항목만 점검한다
  시스템 구조에서 발생하는 신뢰 경계 문제는 잡지 못한다
```

## ✨ 올바른 접근 (After — DFD 기반 위협 모델링)

```
설계 단계에서 DFD를 그리고 신뢰 경계를 식별:

1. 시스템의 모든 데이터 흐름을 그린다
   사용자 → API Gateway → Spring Service → DB
   사용자 → API Gateway → Spring Service → Redis
   Spring Service → 외부 결제 API
   배치 서비스 → DB → S3

2. 신뢰 경계를 표시한다 (점선으로)
   인터넷 ↔ API Gateway (신뢰 경계 1)
   API Gateway ↔ 내부 서비스 (신뢰 경계 2)
   내부 서비스 ↔ DB (신뢰 경계 3)

3. 신뢰 경계를 넘는 모든 데이터 흐름에 STRIDE 적용
   → 경계를 넘는 곳이 공격 표면

결과:
  내부 서비스 간 통신도 인증 필요 → mTLS 도입
  배치 서비스 DB 계정 분리 → 읽기 전용 계정 사용
  S3 버킷 퍼블릭 설정 제거 → Pre-signed URL 방식 전환
```

## 🔬 DFD — Data Flow Diagram 심층 분석

### DFD의 4가지 구성 요소

```
1. 외부 엔터티 (External Entity) — 사각형
   시스템 외부에서 데이터를 주고받는 주체
   예: 사용자(브라우저), 외부 결제 API, 파트너사

2. 프로세스 (Process) — 원형
   데이터를 처리하는 컴포넌트
   예: Spring Service, 배치 잡, Lambda

3. 데이터 저장소 (Data Store) — 평행선 두 줄
   데이터가 저장되는 곳
   예: MySQL, Redis, S3, 로컬 파일

4. 데이터 흐름 (Data Flow) — 화살표
   컴포넌트 간 데이터 이동
   화살표 위에 데이터 유형 명시 (예: "JWT + 주문 요청")
```

### 신뢰 경계 (Trust Boundary)

DFD에서 가장 중요한 개념은 **신뢰 경계**다. 서로 다른 신뢰 수준을 가진 두 컴포넌트 사이의 경계를 점선으로 표시한다. **신뢰 경계를 넘는 모든 데이터 흐름이 위협 분석의 대상**이다.

```
전형적인 Spring 마이크로서비스의 신뢰 경계:

[인터넷] ─────────────── 신뢰 경계 1 ──────────────
  사용자 브라우저
  모바일 앱
  파트너 API 클라이언트
[내부 네트워크] ──────── 신뢰 경계 2 ──────────────
  API Gateway
  Order Service
  User Service
  Payment Service
[데이터 레이어] ──────── 신뢰 경계 3 ──────────────
  MySQL (Primary)
  Redis
  S3
[외부 서비스] ─────────── 신뢰 경계 4 ──────────────
  PG사 결제 API
  SMS 발송 API
  이메일 서비스

분석 원칙:
  경계 1 → 2: 모든 요청을 신뢰하지 않음 (인증/인가 필수)
  경계 2 → 3: 최소 권한 계정, 연결 암호화
  경계 2 → 4: 외부 서비스 응답을 신뢰하지 않음 (SSRF 방어)
  경계 4 → 2: 콜백 요청 서명 검증 (웹훅 보안)
```

### 실제 DFD 위협 분석 예시

```
데이터 흐름: 사용자 → Order Service → DB (주문 생성)
신뢰 경계 통과: 인터넷 → 내부 (경계 1), 내부 → DB (경계 3)

경계 1에서의 위협:
  S: 토큰 위조로 다른 사용자인 척
  T: 주문 금액 변조
  I: 에러에 내부 정보 포함
  D: Rate Limiting 없는 주문 폭탄

경계 3에서의 위협:
  T: SQL Injection으로 DB 데이터 변조
  I: SELECT 권한 초과로 다른 테이블 접근
  D: 대량 쿼리로 DB 부하
  E: DBA 권한 실행으로 스키마 변경
```

## 🔬 PASTA — 7단계 위협 모델링 방법론

PASTA(Process for Attack Simulation and Threat Analysis)는 공격자의 실제 공격 시나리오를 시뮬레이션하여 위험을 평가하는 방법론이다. STRIDE가 위협을 분류한다면, PASTA는 비즈니스 영향까지 연결하여 우선순위를 결정한다.

```
Stage 1: 비즈니스 목표 정의
  "이 시스템에서 가장 비즈니스 영향이 큰 데이터/기능은 무엇인가?"
  예: 결제 데이터, 사용자 개인정보, 주문 처리 기능

Stage 2: 기술 범위 정의
  시스템을 구성하는 모든 컴포넌트 목록화
  Spring Service, DB, Redis, 외부 API, 배포 환경

Stage 3: 애플리케이션 분해 (DFD 작성)
  데이터 흐름 다이어그램 작성
  신뢰 경계 식별
  진입점과 출구점 목록화

Stage 4: 위협 분석
  공격자 프로파일 정의 (동기, 능력, 리소스)
  알려진 공격 패턴과 CVE 연결
  STRIDE로 각 데이터 흐름 분석

Stage 5: 취약점 분석
  실제 코드/설정에서 취약점 식별
  SAST 결과, 의존성 CVE, 설정 오류

Stage 6: 공격 시나리오 시뮬레이션
  구체적인 공격 흐름 작성
  "공격자는 어떤 단계를 거쳐 목표에 도달하는가?"
  예: 피싱으로 직원 토큰 탈취 → 내부 API 접근 → DB 데이터 추출

Stage 7: 위험 및 대응 분석
  각 공격 시나리오의 위험도 정량화
  대응 방안의 비용 대비 효과 계산
  우선순위 기반 로드맵 작성
```

## 📊 위험 우선순위 평가 — 정량화 방법

위협이 많을 때 어디서부터 시작할지 결정하려면 정량적 평가가 필요하다.

### DREAD 모델

```
각 위협에 대해 1~10 점수 부여:

D — Damage (피해 규모)
  1: 개별 사용자 영향
  5: 수천 사용자 영향
  10: 전체 서비스 데이터 유출

R — Reproducibility (재현 가능성)
  1: 특수 조건에서만 가능
  5: 인증된 사용자라면 가능
  10: 누구나 항상 가능

E — Exploitability (공격 난이도)
  1: 고도의 전문 기술 필요
  5: 공개 도구 필요
  10: 브라우저만으로 가능

A — Affected Users (영향받는 사용자 수)
  1: 특정 사용자 1명
  5: 특정 그룹
  10: 전체 사용자

D — Discoverability (취약점 발견 난이도)
  1: 소스코드 접근 필요
  5: 반복 시도로 발견
  10: API 문서만 봐도 알 수 있음

위험 점수 = (D + R + E + A + D) / 5

예시: IDOR 취약점
  D=7, R=9, E=8, A=6, D=8 → 위험도 = 7.6/10 → 즉시 수정
```

### 우선순위 매트릭스

```
           가능성 낮음    가능성 높음
영향 높음  | 주시 (2순위) | 즉시 수정 (1순위) |
영향 낮음  | 수용 (4순위) | 장기 대응 (3순위) |

즉시 수정 (1순위) 예시:
  - 인증 없이 접근 가능한 관리자 API
  - JWT alg:none 공격 가능
  - SQL Injection 가능한 엔드포인트

주시 (2순위) 예시:
  - 물리적 접근이 필요한 취약점
  - 매우 특수한 조건에서만 발생하는 Race Condition

장기 대응 (3순위) 예시:
  - Rate Limiting 없는 비민감 API
  - 서버 버전 정보 노출

수용 (4순위) 예시:
  - 공개된 정보에 대한 정보 열거
```

## 💻 실전 실습 — Sprint에 위협 모델링 통합하기

```
권장 프로세스:

1. 기능 설계 시 (1~2시간)
   새 기능의 DFD를 간단히 그린다 (화이트보드 수준)
   신뢰 경계를 표시하고 STRIDE로 각 흐름 점검
   발견된 위협을 백로그에 보안 태스크로 추가

2. 코드 리뷰 시 (PR마다)
   보안 체크리스트를 PR 템플릿에 포함
   □ 새 엔드포인트에 인증/인가 적용?
   □ 외부 입력값 검증?
   □ 에러 응답 정보 최소화?

3. 분기별 전체 시스템 검토 (반나절)
   전체 DFD 최신화
   신규 컴포넌트/외부 서비스 추가된 신뢰 경계 점검
   이전 분기 위협 대응 결과 검토

위협 모델 문서 보관:
  threat-model/
    dfd-v2.drawio          # 최신 DFD
    threats-2024-Q1.md     # 분기별 위협 목록과 대응 상태
    decisions.md           # 수용한 위험과 이유
```

## ⚖️ 트레이드오프

| 방식 | 장점 | 단점 |
|------|------|------|
| 체크리스트 방식 | 빠르고 일관성 있음 | 알려진 항목만 점검, 구조적 취약점 놓침 |
| DFD 기반 | 시스템 구조에서 위협 발견 | 초기 학습 비용, DFD 작성 시간 |
| PASTA 전체 | 비즈니스 영향까지 연결, 우선순위 명확 | 시간 소요 (반나절~하루), 전문가 필요 |
| 경량 PASTA | 핵심 단계만 적용 (Stage 3~5) | 완전한 비즈니스 영향 분석 미흡 |

소규모 스타트업이라면 DFD + STRIDE 조합으로 시작하여 시스템이 복잡해질수록 PASTA를 도입하는 것이 현실적이다.

## 📌 핵심 정리

- **DFD의 핵심은 신뢰 경계** — 경계를 넘는 모든 데이터 흐름이 위협 분석 대상이다
- **PASTA는 비즈니스 영향을 연결** — 기술 취약점이 비즈니스에 어떤 피해를 주는지까지 평가한다
- **정량화로 우선순위 결정** — DREAD 모델이나 위험 매트릭스로 어디서부터 시작할지 결정한다
- **스프린트에 통합** — 별도의 "보안 스프린트"가 아니라 설계/리뷰 단계에서 자연스럽게 수행한다
- **문서를 남긴다** — 수용한 위험과 이유를 기록해야 나중에 같은 논의를 반복하지 않는다

## 🤔 생각해볼 문제

**문제 1**: 다음 마이크로서비스 구성에서 신뢰 경계를 표시하고, 각 경계에서 가장 위험한 위협 하나씩을 찾아라.

```
구성:
  사용자 → Nginx (80/443) → Spring Gateway → Order Service
  Order Service → MySQL
  Order Service → Redis
  Order Service → 외부 배송 추적 API (HTTP 콜백 수신)
```

**해설**:
```
신뢰 경계:
  경계 1: 인터넷 ↔ Nginx
  경계 2: Nginx ↔ Spring Gateway
  경계 3: Spring Gateway ↔ Order Service
  경계 4: Order Service ↔ MySQL
  경계 5: Order Service ↔ 외부 배송 API

각 경계의 주요 위협:
  경계 1: DDoS, SSL Strip (HSTS 미설정)
  경계 2: Gateway 우회 (내부 포트 직접 접근 시)
  경계 3: 서비스 간 인증 없음 (내부 네트워크 신뢰 가정)
  경계 4: SQL Injection, 과도한 DB 권한
  경계 5: 배송 API 콜백 서명 미검증 → 위조 배송 상태 주입 (SSRF의 역방향)
```

**문제 2**: PASTA Stage 6(공격 시나리오 시뮬레이션)에서 "외부 공격자가 고객 결제 정보를 탈취하는 시나리오"를 구체적인 단계로 작성해보라.

**해설**:
```
공격 시나리오:
  Step 1: 정찰 (Reconnaissance)
    - 공개된 API 문서/Swagger UI에서 엔드포인트 목록 수집
    - 에러 응답에서 Spring Boot 버전 파악
    - 알려진 CVE 검색

  Step 2: 진입점 탐색
    - 로그인 API Rate Limiting 없음 확인
    - 브루트포스로 테스트 계정 탈취

  Step 3: 권한 상승
    - 탈취한 계정으로 다른 사용자 주문 목록 조회 (IDOR)
    - 결제 API 엔드포인트에서 카드번호 응답 발견

  Step 4: 데이터 유출
    - 자동화 스크립트로 전체 주문 데이터 수집
    - 응답에 카드번호 마스킹 없이 포함됨

대응:
  - 로그인 Rate Limiting
  - 주문 조회 시 소유자 검증
  - 카드번호 응답 마스킹 (앞 12자리 마스킹)
  - 이상 탐지 (짧은 시간에 다수 주문 조회)
```
